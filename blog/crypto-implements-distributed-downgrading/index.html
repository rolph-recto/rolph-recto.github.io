Cryptography Implements Distributed Downgrading

Jul 09, 2021

One of the main challenges in ensuring security in the distributed setting is the fact that the machines executing the program only have the partial trust of the principals involved. For example, even if the users of my web application trust me with their private data, they might not confer the same trust to the third-party service that my application calls out to. But say the third-party service really, really needs to use my users’ private data to compute some result that my application needs. Is there a way to let the service compute over my users’ data without actually revealing it?

Yes! In the particular case above, we can use homomorphic encryption to allow the third-party service and let it perform computations with my users’ encrypted data without revealing the actual values of the data. In general, cryptography is one of the most powerful solutions to the problem of ensuring security under partial trust. However, I must stress here that crypto is a powerful tool for implementing security policies, but it says nothing about specifying security policies.

In my lab, there has been a longstanding folklore principle about ensuring security for distributed programs: the use of cryptography is usually required at downgrading boundaries. In this post, I’m going to explain what this means.

Recall that downgrading means that we either make information less secret (downgrading along the confidentiality axis) or we make information more trusted (downgrading along the integrity axis). Downgrading is so named because if we take the standard lattice-based model for information flow, it is an action that moves information from a security label higher in the lattice to a security label lower in the lattice. This of course violates the principle that information only “flows up” the lattice—public information can flow to secret information but never secret to public; trusted information can flow to untrusted information but never untrusted to trusted. Information only “flows up” the lattice for programs that satisfy noninterference; more sophisticated security conditions such as nonmalleable information flow are needed to ensure that downgrading is used securely.

In practice, noninterference is almost never the security condition that you actually want for distributed programs, as downgrading is almost always necessary in distributed programs.1 For example, a distributed program might want to collect data from many different principals and make public (read: declassify) some computed aggregate data from this. A program might require a machine controlled by one principal to provide some input, and then prevent further modification so that other principals can trust its value (read: endorse the input).

The important feature that distinguishes ensuring security for distributed programs over other kinds of programs is that the machines executing the program can have only the partial trust of principals involved. For example, take the classic millionaires’ game: Alice and Bob want to determine who is richer but neither wants the other to figure out their exact net worth. The dilemma, then, is how to perform the comparison when Alice’s machine is not trusted by Bob with his private data, and vice versa. How can we then implement this distributed program?

If Alice and Bob both trust a machine—a trusted third-party—enough to store their private data, then the dilemma can be solved, as they just need to pass their data over to the trusted third-party to perform the comparison and then send back the result to them. Noninterference is the standard security condition defended in language-based security, but in practice it’s almost never the security condition that you really want. This is particularly true in the distributed setting, where machines are only partially trusted by principals. To recap: for a program to satisfy nonintereference, secret data must not influence public data, and untrusted information must not influence trusted data. So for a distributed program to satisfy nonintereference, it must be the case that the following hold:

  • Only machines that you trust with your private data can operate on it.

  • Only machines that you trust with your private data can operate on it.

  1. I’m going to go out on a limb here and say that noninterference is almost useless as a practical security condition.
