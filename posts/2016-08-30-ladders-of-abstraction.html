Throwing Away the Ladder of Abstraction

August 30, 2016 - compsci, haskell

The Tractatus Logico-Philosophicus is hard to understand, as anyone who has tried to read it will tell you. It is terse to a fault, with Wittgenstein unwilling to spare any exposition whatsoever to explain his weird ideas. But I think it is this terseness that lends the Tractatus a mystical, poetic quality to it. It is hard to understand, yes, but a lot of fun to read.

One of the most famous images from the Tractatus occurs in its penultimate section 1:

He must, so to speak, throw away the ladder after he has climbed up it.

In this context Wittgenstein is quite remarkably advising the reader to regard what he has written up to now as being “nonsensical,” and that to truly reach the conclusion that he is pointing the reader to, she must forget how she got there. Quite loopy, yes? 2

I’ll skip the philosophical discussion for now and instead co-opt the image to explain something else entirely. While he didn’t have it in mind when he wrote the Tractatus, I think one can read from it good advice concerning programming.

Haskell is incredibly malleable and allows you to express a solution elegantly and succintly. This is perhaps the biggest reason why people tolerate its relatively steep learning curve. Instead of moulding a solution to fit Haskell, I find instead that often times I can mould Haskell to the problem.

The claim that Haskell is “the best imperative language” is only tongue-in-cheek up to a certain point.

Imperative programs have “semantic locality.” It is easy to understand what is going on at any section of code. The problem is that, combined together, these small sections being so unconstrained at what they can do, it becomes hard to keep track of what your program does.

In Haskell, often the program is a lot shorter, and so easier to understand as a whole. But how small pieces of it work is often harder to understand. Type classes create “semantic distance” by hiding functionality away from the current piece of code.

Here is the implementation of backwards chaining from Russell and Norvig:

# returns a generator of substitutions
def BC-ASK(KB, query):
  return BC-OR(KB, query, {})

# yields a substitution
def BC-OR(KB, goal, t):
  for Rule(lhs, rhs) in FetchRulesForGoal(KB, goal):
    (lhs2, rhs2) = StandardizeVariables(lhs,rhs)
    for t2 in BC-AND(KB, lhs2, Unify(rhs2, goal, t):
      yield t2

# yields a substitution
def BC-AND(KB, goals, t):
  if t == failure: return
  else if Length(goals) == 0: yield t
  else:
    (first, rest) = First(goals), Rest(goals)
    for t2 in BC-OR(KB, Subst(t,first), t):
      for t3 in BC-AND(KB, rest, t2):
        yield t3

Now here is a Haskell implementation:

type QueryM a = ListT (State (M.Map T.Text Int)) a

backwardChainOr :: Pred -> Sub -> QueryM Sub
backwardChainOr goal sub = do
  let goalRules = getGoalRules goal
  goalRule <- getGoalRules goal
  Rule head body <- getFreshVars goalRule
  -- if the head of the rule fails to unify with the 
  -- goal, then we kill this branch of the search path.
  let sub' = unifyNoOccurs head goal sub
  backwardChainAnd body sub'

backwardChainAnd :: [Pred] -> Maybe Sub -> QueryM Sub
backwardChainAnd goals msub
  -- if m is a MonadPlus, then StateM s m a is a MonadPlus also
  -- in this case, the mzero for list is [], which effectively
  -- kills this branch of the search path.
  | Nothing <- msub                 = do
    mzero

  | Just sub <- msub, [] <- goals   = do
    return sub

  | Just sub <- msub, g:gs <- goals = do
    -- we try to find a proof for the head
    let g' = applySubPred sub g
    sub' <- backwardChainOr g' sub
    -- then we try to find proofs for the rest of preds in the tail
    backwardChainAnd gs (Just sub')

The line where the result of getGoalRules goal is bound to goalRule really means something like

fork off a stateful computation that takes one of the possible rules that is compatible with the goal predicate and try to find a proof using that rule

Needless to say, expressing this would be a bit more verbose in other languages.

The computation can possibly have side effects because it can change the supply of fresh variables through getFreshVars, which works by appending a counter to a name. The counter is then incremented so a never before seen instance of the name can be generated next time. This guarantee might fail when the program is processing a massive query and overflow occurs.
