Partial Application

Partial application, if you’re unfamiliar with the term, is the ability to apply only some of the arguments of a function. Instead of returning the result of the function, a partially applied function returns a new function with fewer arguments. For example, the plus function (+) requires two arguments and returns a number; but when partially applied as in [+ 1] it returns a new function that requires one argument and returns a number. Partial application for our interpreter is quite easy to implement.

Partial application is automatically supported for user-defined functions, but its implementation for primitive functions is slightly complicated. Consider what would happen if we partially applied the argument to the if primitive function, like so:

[if [== 1 2] [print "truth!"]]

Obviously the conditional above will always take its else branch, which corresponds to its third argument. But there is no third argument! On the other hand, the primitive function (+) when partially applied has a straightforward value, as discussed above. So it looks like partial application makes sense for some primitive functions but not others.

What’s going on? The problem is that primitive functions really represent two pieces of the language: they can represent both expression forms and control structures. For expression forms like +, there is still a clear meaning to the form when it is partially applied. But for control structures like if, partial application does not really make any sense.

One way to solve the problem is to have two kinds of primitive functions that reflect the difference between expression forms and control structures. However, I’ve found that the simplest solution, in terms of implementation, is disallow partialy application for primitive functions altogether. Primitive functions that are expression forms are then wrapped in user-defined functions to allow partial application. For example, the primitive plus function would be named __PRIM__+, and then a user-defined plus function + would be defined that calls the primitive plus function, like so:

[fun + [x y] [__PRIM__+ x y]]

The only problem with this pollutes the environment with ugly bindings. I think this is an acceptable tradeoff for the simplicity of the solution.
